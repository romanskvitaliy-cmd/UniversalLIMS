# Етап 2b: Конструктор шаблонів на базі PDF + Overlay

## Мета

Перейти від `.docx`/Word-орієнтованого підходу до єдиного формату `PDF + Overlay`, щоб гарантувати pixel-perfect незмінність державних бланків і стабільну роботу в браузері.

## Чому обрано PDF + Overlay

- Юридично критична верстка державних форм не повинна змінюватися.
- Один формат для перегляду, заповнення і видачі зменшує ризики та складність.
- Поля і правила зберігаються в БД, а PDF є візуальною основою.
- Простіше реалізувати жорсткі обмеження довжини і field-level security.

## Архітектурні принципи

- Immutable by design: фізичне видалення заборонено, тільки анулювання з причиною.
- Повний Audit Trail для всіх адміністративних дій.
- Версійність шаблонів: виданий документ назавжди прив'язаний до версії.
- Field-level security: `Edit` / `ReadOnly` / `Hidden` для кожного поля і ролі.
- Single source of truth: значення полів, обмеження і права живуть у БД.
- Pixel-perfect: PDF є незмінним візуальним каркасом.

## Цільова модель роботи

1. Адміністратор завантажує PDF-шаблон.
2. У дизайнері бачить сторінки PDF у браузері.
3. Додає поля overlay на конкретні координати сторінки.
4. Прив'язує поле до `DataField` або створює новий `DataField`.
5. Налаштовує місткість, multiline та overflow-політику.
6. Налаштовує права доступу на поле для ролей.
7. Публікує версію після проходження валідації.

## Що перевикористовуємо з поточного коду

- `Template`, `TemplateVersion`, `TemplateField`, `TemplateFieldPermission`.
- `DataField` та бібліотеку тегів.
- Soft annulment, audit trail, версійність, policy-авторизацію.
- Локальне сховище документів і контроль SHA-256.

## Що виводимо з active-реалізації

- Автоскан `.docx` через `IDocxContentControlReader`/`ZipDocxContentControlReader`.
- Word-специфічний UX: `ms-word:` відкриття та token-flow для Word desktop.
- Жорстку залежність процесу від Content Controls у Word.

## Нові технічні вимоги до конструктора

- PDF preview у браузері (наприклад, через pdf.js або еквівалент).
- Overlay-шар з координатною моделлю: `PageNumber`, `PositionX`, `PositionY`, `Width`, `Height`.
- Редактор властивостей поля в сайдбарі.
- Візуальна і серверна валідація MaxLength/Capacity перед публікацією.
- Блокуючі правила publish: немає unmapped required полів, немає дірок у permission matrix.

## План міграції

### Крок 1: Підготовка доменної і сервісної межі

- Додати формат шаблону на рівні версії (`Pdf` / `DocxLegacy`).
- Перевести створення полів на сервіс дизайнера overlay для `Pdf`.
- Ізолювати Word-функції за умовою `DocxLegacy`.

### Крок 2: UI PDF Overlay Designer

- Додати сторінку конструктора з preview PDF та drag/resize полів.
- Додати CRUD для координат і властивостей поля.
- Додати масове редагування permission matrix.

### Крок 3: Публікація і валідація

- Оновити validator під PDF-модель.
- Заборонити publish без геометрії та мапінгу required полів.
- Перевіряти конфлікти/накладання полів за правилами, погодженими з бізнесом.

### Крок 4: Депрекація Word-контуру

- Прибрати Word-кнопки з активного UI для нових версій.
- Лишити читання legacy даних тільки для історії.
- Після стабілізації виконати міграцію БД для видалення Word-специфічних залишків.

## Критерії готовності

- Адміністратор повністю розмічає новий бланк без Word.
- Візуальний результат у системі збігається з виданим PDF.
- Field-level security і MaxLength працюють на рівні кожного поля.
- Всі зміни трасуються в аудиті, анулювання тільки з причиною.
