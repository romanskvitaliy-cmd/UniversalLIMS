# Філії, робочі місця (5–10 ПК) і сповіщення

> Оновлено: 2026-05-30  
> Мета: не ламати поточну модель `Branch` / `Order` / `TargetBranch`, але зрозуміло пояснити адміну й керівництву.

---

## 1. Що вже є в проєкті (не чіпати ядро)

| Поняття | Де в БД | Навіщо |
|---------|---------|--------|
| **Філія лабораторії** | `Branch`, `OrderDocument.TargetBranchId` | Куди пішла проба на аналіз (Житомир, Бердичів, Корosten…) |
| **Філія реєстратури** | `Order.BranchId`, `ApplicationUser.BranchId` | Де прийняли справу; фільтр реєстру та видачі |
| **Контекст лабораторії** | Session `ActiveLaboratoryBranchId` (адмін) або claim `BranchId` (лаборант) | Журнал і PDF бачать «свою» лабораторію |
| **Сповіщення** | Poll API + `lims-route-notify.js` | Toast/голос на кожному ПК окремо |

**Висновок:** окрема сутність «Робоче місце №3» **не потрібна** для v1. П’ять–десять комп’ютерів в одному кабінеті = **одна філія**, один черговий реєстр/журнал, сповіщення на всіх моніторах (це плюс для залу прийому).

---

## 2. Два сценарії (як зручніше)

### Сценарій A — один кабінет, 5–10 місць (найчастіший)

- Адмін створює **одну** філію реєстратури (наприклад `REG-ZHY` «Реєстратура Житомир»).
- Усі реєстратори мають `BranchId` цієї філії (або спільний обліковий запис — гірше для аудиту).
- Усі ПК показують **той самий** `/Orders` і `/Issuance`; poll-сповіщення на кожному браузері.
- **Нічого додавати в код**, окрім увімкнених сповіщень.

### Сценарій B — кілька пунктів прийому (як лабораторії)

- У довіднику **Філії** (`/Branches`) адмін заводить:
  - лабораторії: `LAB-BER`, `LAB-KOR`…
  - реєстратури: `REG-ZHY`, `REG-BER`…
- Кожному користувачу в `/Users` — свій `BranchId`.
- `Order.BranchId` = філія реєстратора при створенні (вже так).
- `OrderDocument.TargetBranchId` = лабораторія виконання (вже так).
- Реєстратор Бердичева **не бачить** справи Житомира (фільтр по `Order.BranchId`).

**Рекомендація ЦКПХ:** сценарій B, якщо реєстратура фізично в кількох містах; сценарій A, якщо один зал з багатьма вікнами.

---

## 3. Ланцюжок сповіщень (обов’язковий цикл)

| Крок | Хто отримує | Подія в системі |
|------|-------------|-----------------|
| Реєстратор → лабораторія | Лаборант | `RoutedAtUtc` / відправка документів |
| Лаборант → експерт | Експерт | `Sample.ResultsEnteredAtUtc` (усі шаблони «Відправити експерту») |
| Експерт → доопрацювання | Лаборант | `ExpertConclusionReview.ReturnedForReworkAtUtc` |
| Експерт затвердив | Реєстратор | `Sample.ReadyForPickupAtUtc` |
| Видача клієнту | — | `IssuedAtUtc` (архів, без poll) |

Технічно: один JS-модуль poll, різні API за роллю (`/api/laboratory/...`, `/api/expert/...`, `/api/registration/...`).

---

## 4. Передача лабораторія → експерт (процес)

1. Лаборант заповнює PDF (усі обов’язкові поля).
2. Натискає **«Відправити експерту»** (toolbar PDF або список документів проби).
3. `LaboratoryDocumentSubmissionService` ставить документ `ResultsEntered`; коли **всі** документи проби готові — `Sample.ResultsEntered` + timestamp для сповіщення експерта.
4. Експерт бачить пробу в `/Expert` і отримує toast.

Без кроку 2 проба **не потрапляє** в чергу експерта (навмисно — захист від недозаповнених бланків).

---

## 5. Що адміністратор налаштовує

1. **Філії** — `/Branches` (коди, міста, активність).
2. **Користувачі** — `/Users` (роль + `BranchId`).
3. **Шаблони** — прив’язка до типів досліджень.
4. На пілоті: перевірити, що лаборанти й реєстратори мають правильний `BranchId`.

Окремий екран «10 місць» **не планується** — місця = користувачі або філії.

---

## 6. Що можна додати пізніше (не v1)

- Тип філії `Registration` / `Laboratory` у UI (зараз розрізняють кодом/назвою).
- Контекст філії реєстратора в session (як у адміна для лабораторії) для супер-користувача.
- Призначення проби конкретному лаборанту (зараз — спільна черга філії).
- SignalR замість poll (якщо мережа навантажена).
